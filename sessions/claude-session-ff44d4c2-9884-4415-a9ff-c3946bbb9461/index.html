Critical Pitfalls

Pitfall 1: The Judge Is Grading Its Own Output (Self-Reference Collapse)

What goes wrong: When the same model serves as agent, judge, and optimiser, the judge develops systematic blind spots aligned with how the agent reasons. It scores agent outputs highly not because they are correct but because they mirror its own reasoning style and vocabulary.

Why it happens: Same model means same prior distributions over language. The judge's rubric interpretation tracks the agent's output framing rather than ground truth. For judgment items ("is this clause commercially significant?"), the model's prior about significance dominates over rubric instructions.

Consequences: The experiment appears to "work" — scores converge — but convergence reflects model self-consistency, not extraction quality. The thesis (extraction converges, judgment plateaus) may appear validated for the wrong reason.

Warning signs: - Judge scores for judgment items increase at the same rate as extraction items in early iterations. - Judge reasoning paraphrases the agent output rather than evaluating against rubric criteria. - The pre-written "flawed" review scores higher than expected.

Prevention: Run the pre-loop judge test with outputs that are deliberately bad in ways the agent would not produce. Log the judge's reasoning text (not just scores) for every iteration — if reasoning becomes more similar to agent output over iterations, self-reference is occurring. Accept this as a known confound and document it; do not pretend it doesn't exist.

Phase: Pre-loop judge test must catch this before the main loop runs. This is also a data analysis concern — design logging to capture it.

Confidence: HIGH — well-documented in Zheng et al. (2023) "Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena."

Pitfall 2: Structured Output Schema Validates Structure, Not Meaning

What goes wrong: Ollama's JSON mode forces parseable output, but the content inside valid fields can be semantically empty, wrong, or hallucinated. Pydantic validates structure, not meaning. A score of 7 with reasoning "The agent correctly identified the issue" is syntactically valid but evidentially worthless.

Consequences: The experiment produces clean, parseable output for every iteration with no indication the judge is making up scores. Analysis downstream treats fabricated scores as evidence.

Warning signs: - Reasoning strings are short (under 40 words) across many items. - Reasoning uses identical phrasing across rubric items ("The agent addressed this criterion adequately"). - Scores cluster at round numbers (5, 7, 8) with no variance across items that should differ.

Prevention: Add content validators to Pydantic models: score must be 0–10, reasoning must exceed N characters (e.g., 50), reasoning must reference rubric criterion by name or keyword. Treat iteration 1's judge output as a calibration run — manually inspect all 8 rubric item scores and reasoning before trusting automated scores. Log raw model output before Pydantic parsing.

Phase: Judge component build. Validator logic should be in the Pydantic model, not bolted on later.

Confidence: HIGH — fundamental property of schema validation.

Pitfall 3: Ollama Temperature=0 Is Not Deterministic Across Runs

What goes wrong: Temperature 0 reduces but does not eliminate variance. Different hardware, different context lengths, and GGUF quantisation precision all affect floating-point operations. Two runs with temperature=0 on the same input can produce different outputs.

Why it happens: Temperature controls sampling stochasticity, but transformer floating-point arithmetic is subject to hardware non-determinism. GGUF quantised models amplify this because rounding at each quantisation step interacts with accumulated floating-point errors.

Consequences: If you re-run the experiment expecting identical results, you may not get them. The pre-loop judge test comparison (good vs flawed review) may flip on re-run.

Warning signs: - Pre-loop judge test gives different scores on consecutive runs. - Agent output for identical prompts differs across runs.

Prevention: Do not claim reproducibility across machines — only within the same machine and model file. Log the Ollama model name including quantisation tag (e.g., qwen2.5:7b-q4_K_M, not just qwen2.5:7b). Record the Ollama version (ollama --version) in experiment metadata. Run the pre-loop judge test twice consecutively; if results differ, treat all scores as having ±1 variance.

Phase: Experiment setup / configuration. Add to config.json or experiment metadata immediately.

Confidence: MEDIUM — observed in practice; Ollama-specific behaviour may vary by version.

Pitfall 4: OpenAI SDK + Ollama Structured Output Schema Incompatibility

What goes wrong: Ollama's OpenAI-compatible endpoint does not support all response_format features the OpenAI SDK exposes. response_format={"type": "json_schema", "json_schema": {...}} (strict mode, introduced late 2024) may silently fall back to unstructured output or raise an error depending on the Ollama version. The OpenAI Python SDK's client.beta.chat.completions.parse() method may not work with Ollama endpoints.

Why it happens: Ollama implements the OpenAI API surface incrementally. The SDK may send parameters Ollama ignores without raising an error.

Consequences: You write code using client.beta.chat.completions.parse() expecting Pydantic integration, but Ollama silently ignores the schema and returns plain text. The retry loop then fires on every call, creating 3x the expected latency.

Warning signs: - Judge returns natural-language text when you expect JSON. - Retry count is always hitting the maximum (3) in early testing. - Model output contains JSON embedded in prose ("Here is the evaluation: {...}") rather than raw JSON.

Confidence: MEDIUM — based on Ollama changelog through mid-2025; verify with current Ollama version before building.

Pitfall 5: The Optimiser Rewrites Into the Model's Comfort Zone, Not Toward Correctness

What goes wrong: The optimiser receives only the prompt and judge feedback. Without seeing the NDA, it can only rewrite the system prompt to produce outputs the judge will score higher. Consequences: By iteration 5–8, the agent prompt may instruct the agent to mirror rubric language explicitly, which boosts scores without improving analysis quality.

Warning signs: - Agent system prompt begins to contain phrases from the rubric (e.g., "assess whether the clause is commercially significant"). - Extraction scores and judgment scores rise in lockstep after iteration 3. - Judge feedback stops mentioning specific deficiencies and becomes generic praise.

The pre-written "good" and "flawed" reviews serve as a fixed reference that the optimiser cannot touch — use them as a sanity check at the end.

What goes wrong: The judge receives: NDA text + agent output + rubric (8 items) + playbook + judge instructions. For a 7B or 13B local model, this can easily exceed the effective context window. Ollama truncates silently — it does not raise an error; it simply drops content from the beginning of the context.

Why it happens: Ollama's num_ctx parameter defaults to 2048 in some versions unless explicitly overridden. Local models also have stated context lengths (32k, 128k) that perform poorly beyond 8–16k tokens in practice.

Consequences: The judge silently receives truncated input, scores without seeing the full NDA or full agent output, and produces scores that appear valid. Only detectable by manually reviewing judge reasoning that references content that wasn't in its truncated context.

Warning signs: - Judge reasoning refers to "the NDA" generically without citing specific clause numbers or text. - Judge gives the same scores across different agent outputs that vary in quality. - Ollama response latency is suspiciously low (model truncated most input).

Prevention: - Explicitly set num_ctx in Ollama model options on every API call: options={"num_ctx": 8192}. Do not rely on the default. - Estimate token counts before running the loop: 1 token ≈ 0.75 words. Ensure all judge inputs fit within 70% of the configured context window. - Keep the synthetic NDA short (under 1500 words) — enough for 4 embedded issues but not so long it dominates context. - Log the total character count of the judge's prompt at iteration 1.

Phase: Judge component build. Must set num_ctx before first test run.

Confidence: HIGH — Ollama's default num_ctx of 2048 is a well-documented common gotcha.

Pitfall 7: Retry Loop Masks Systematic Failure, Not Transient Failure

What goes wrong: The 3-attempt Pydantic retry loop assumes transient JSON malformation. If the model cannot produce valid JSON for a given input at all, all 3 retries fail in the same way. An undefined failure path (raise exception, return None, return empty scores) corrupts the experiment log silently.

Consequences: Experiment log contains null scores for some iterations, confounding downstream analysis. Or the loop crashes at iteration 4 and the experiment is incomplete.

Warning signs: - All retries fail for the same rubric item consistently. - Exception occurs at iteration N but not N-1 (prompt grew longer; context overflow is the real cause).

Prevention: - Define explicit failure handling: on 3-retry exhaustion, log the raw output, log an error entry in the JSON results, and continue to the next iteration — do not crash. The loop must be resilient. - Add a validation_attempts field to the judge output schema so you can see how many retries each call required. - On retry, append a correction prompt ("Your response was not valid JSON. Return only a JSON object matching this schema: {schema}") rather than repeating the identical prompt.

Phase: Judge component build. Failure handling must be designed before the retry loop, not after.

Confidence: HIGH — standard engineering concern.

Pitfall 8: Evaluation Contamination — Rubric Vocabulary in the Agent Prompt

What goes wrong: If the agent system prompt includes rubric criterion language or playbook language verbatim, the agent produces outputs that mirror rubric phrasing. The judge, scoring against the rubric, gives high scores because the vocabulary matches. This is echo-chamber scoring, not valid evaluation.

Consequences: High extraction AND judgment scores from iteration 1 with no convergence pattern. The thesis cannot be tested because contamination removes the extraction/judgment distinction.

Warning signs: - Agent output uses rubric phrases like "I note this is commercially significant" in iteration 1 without optimiser feedback having introduced that framing. - Scores are uniformly high from iteration 1 with no optimisation signal.

Prevention: - The agent system prompt must NOT reference the rubric, playbook, or evaluation criteria. - The agent prompt should describe the task in domain terms only: "Review this NDA for issues relevant to [party type]. For each issue, identify the clause, extract the relevant text, and provide your assessment." - Include an explicit instruction in the optimiser: "Do not reveal to the agent that it is being evaluated, or share evaluation criteria with it." - Inspect iteration 1's agent system prompt against the rubric for vocabulary overlap before running the full loop.

Phase: Agent component build. Prompt design review before first run.

Confidence: HIGH — standard evaluation methodology concern.

Moderate Pitfalls

Pitfall 9: JSON Results Files Growing Unbounded or Corrupting on Interruption

What goes wrong: Each iteration logs substantial JSON. If written as a single file with append semantics, JSON structure breaks on interruption. If written as separate files without a clear naming convention, aggregation becomes messy.

Prevention: Write one JSON file per iteration (e.g., results/iteration_001.json) plus a results/summary.json updated after each iteration. Never append to a JSON array file without load-append-rewrite; or use JSON Lines (one object per line, newline-delimited) which is append-safe. Test interruption recovery before the main experiment run.

Phase: Main loop.

Pitfall 10: The Pre-Loop Judge Test Is Not Actually Diagnostic

What goes wrong: The test runs a "good" and "flawed" review through the judge. If the score delta is small (good=7.2 average, flawed=6.8 average), the judge is not discriminating and all subsequent loop scores are meaningless. Teams run the test but don't set a minimum acceptable discrimination threshold.

Prevention: Define a minimum acceptable score gap before running the test (e.g., the "good" review should score at least 2.0 points higher on average). If the gap is below threshold, do not proceed to the main loop. Pre-write the "flawed" review with specific, detectable errors — not just vague quality issues.

Phase: Pre-loop judge test (before the main loop begins).

Pitfall 11: Optimiser Prompt Gets Longer Every Iteration

What goes wrong: Without explicit length constraints, each optimiser rewrite tends to add rather than replace — the agent system prompt grows from 200 words to 800 words by iteration 8. Longer prompts stress the agent's context window and change task framing unpredictably.

Prevention: Include a hard length constraint in the optimiser instruction: "Rewrite the system prompt in no more than [N] words." Log prompt word count at each iteration.

Phase: Optimiser component build / Main loop.

Pitfall 12: Pydantic v1 vs v2 API Mismatch

What goes wrong: Pydantic v2 (standard as of 2025) has breaking API changes from v1. model_validate_json(), model_dump(), and field validator syntax differ significantly. Old tutorial code produces runtime errors or silent wrong behaviour.

Prevention: Use uv add pydantic&gt;=2.0 and write all validators using v2 syntax (@field_validator, model_config = ConfigDict(...)). Use model_validate_json(raw_string) not parse_raw(raw_string) (deprecated in v2). Do not mix @validator (v1) and @field_validator (v2) decorators.

Phase: Judge component build.

Pitfall 13: Model Hallucinating Clause Numbers That Don't Exist

What goes wrong: Local models frequently hallucinate specific clause references (e.g., "Clause 4.2 states...") when the NDA does not use numbered clauses. The judge may score it positively because the structure appears precise.

Prevention: Give the synthetic NDA explicit clause numbers and headings. This makes fabricated references detectable. Optionally add a post-processing check that extracts all clause references from agent output and verifies they exist in the NDA.
